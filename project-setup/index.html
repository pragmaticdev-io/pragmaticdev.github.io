Setup a Local Cassandra for Development

Now there are several different ways to setup a Cassandra cluster. I will be configuring my local Cassandra instance using Docker. We can do this using Docker Compose and the official Cassandra Image. We can then specify in our docker-compose.yml the following.

 1
 2
 3
 4
 5

11

version: "3"
services:
  cassandra:
    image: cassandra:3
    ports:
    - "9160:9160"
    - "9042:9042"
    volumes:
    - cassandra-persistence:/var/lib/cassandra
volumes:
  cassandra-persistence:

We can then start it using docker-compose up and connect to it using cqlsh

1
2
3
4
5

> docker run -it --rm --net host cassandra:3 cqlsh
Connected to Test Cluster at 127.0.0.1:9042.
[cqlsh 5.0.1 | Cassandra 3.11.6 | CQL spec 3.4.4 | Native protocol v4]
Use HELP for help.
cqlsh> DESC KEYSPACES

This command may seem a bit odd at first, especially if this is your first time working with Docker, so let me break it down.

  • docker - the docker executable.
  • run - specifying that we want to run a container.
  • -it - specifies we want the launched container to be in interactive mode and we want to attach a TTY session.
  • --rm - a quality of life flag that will destroy the launched container on termination.
  • --net host - set the containers networking to that of the host machine. This way we can connect to other containers through our computer network and not a virtual docker network.
  • cassandra:3 - use the latest image with a Cassandra major version of 3.
  • cqlsh - the command we want to launch in the container.

Setup Cassandra for UnitTests

Update: May 16, 2020

So it looks like the cassandra-unit modules does not work with any Java version after Java 8 (we are using Java 11). 😞 The current solutions are rollback to Java 8 or use the local Cassandra we created for the unit tests. I opted for the latter because Java 8 is reaching End-of-Life at the end of 2020. To configure our local cassandra for unit tests we need to create a keyspace for them to operate in. I created the following keyspace

1

create keyspace untidy_unit_test with replication = {'class': 'org.apache.cassandra.locator.SimpleStrategy', 'replication_factor': '1'};

We can then update our test application.properties to use the separate namesapce and to auto-create the tables required.

spring.data.cassandra.keyspace-name=untidy_unit_test
spring.data.cassandra.schema-action=recreate_drop_unused

While we could use the same Cassandra instance we created above for Unit Testing it will probably cause us more pain than it's worth. By choosing to use our local Cassandra we will run into issues of consistency. The database can be modified during the execution of UnitTests leading to some unexpected errors. Not to mention it makes our code less portable as it will require a local Cassandra to be configured correctly everywhere we want to run our unit tests.

To get around this, we can use an embedded cassandra server. By using an embedded server we have more control over the database and can specify our CQL scripts to run before and after each unit test. Some examples would be for populating test data or recreating a keyspace before the next test is run.

To get started we can add the following dependencies to our build.gradle.kts file.

1
2
3
4

dependencies {
 // omitted other dependencies for cleanliness
	testImplementation("org.cassandraunit:cassandra-unit-spring:3.11.2.0")
}

Cassandra Unit will create an embedded cassandra server for our unit if we mark the tests with some annotations. Some of the annotations provided can be seen further below. It is important to note that this embedded server does not start on the default 9042 port so we need to specify which port to use in our test application.properties file

# src/test/resources/application.properties
spring.data.cassandra.keyspace-name=unitTest
spring.data.cassandra.port=9142

After we have completed that, we can then add the following annotations to our src/test/kotlin/io/pragmaticdev/untidy/UntidyApplicationTests.kt to tell spring to load our embedded cassandra before loading the application context.

 1
 2
 3
 4

10

@CassandraDataSet(keyspace = "unitTest", value = ["cql/dataset.cql"])
@CassandraUnit
@TestExecutionListeners(value = [CassandraUnitDependencyInjectionTestExecutionListener::class, DependencyInjectionTestExecutionListener::class])
@SpringBootTest
class UntidyApplicationTests {

    @Test
    fun contextLoads() {
    }
}

and then create the cql/dataset.cql in our test resources folder with the following data

1
2
3
4

-- src/test/resources/cql/dataset.cql
CREATE KEYSPACE IF NOT EXISTS unitTest
    WITH replication = {'class': 'SimpleStrategy', 'replication_factor': '1'}
        AND durable_writes = true;
